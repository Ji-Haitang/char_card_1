## 一、项目现状分析

### 1.1 当前架构

项目当前作为 SillyTavern（以下简称 ST）的前端卡片运行，核心文件结构：

```
index.html                  # 主页面（含大量 inline JS）
module/
  ├─ game-config.js         # 游戏配置（NPC、地点、数值范围等）
  ├─ game-state.js          # 游戏状态管理（存档/读档）

char_card_information/       # 提示词/人设文本（原本嵌入ST角色卡）
```

### 1.2 对 SillyTavern 的依赖清单

| 依赖项 | 当前实现 | 涉及文件 |

改造的核心目标：**让 index.html 游戏脱离 SillyTavern，以独立网页形式运行，直接对接 LLM API**。

**参考 cs2 的实现：**

**OpenAI 兼容请求格式：**
```javascript
async callOpenAI(messages) {
    const response = await fetch(`${this.config.endpoint}/chat/completions`, {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'Authorization': `Bearer ${this.config.apiKey}`
        },
        body: JSON.stringify({
            model: this.config.model,
            messages: messages,
            temperature: this.config.temperature,
            max_tokens: this.config.maxTokens
        })
    });
    const data = await response.json();
    return data.choices[0].message.content;
}
```

**Gemini 请求格式：**
```javascript
async callGemini(messages, systemPrompt) {
    const url = `${this.config.endpoint}/models/${this.config.model}:generateContent?key=${this.config.apiKey}`;
    const contents = messages.map(m => ({
        role: m.role === 'assistant' ? 'model' : 'user',
        parts: [{ text: m.content }]
    }));
    const response = await fetch(url, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
            contents,
            systemInstruction: { parts: [{ text: systemPrompt }] },
            generationConfig: {
                temperature: this.config.temperature,
                maxOutputTokens: this.config.maxTokens
            }
        })
    });
    const data = await response.json();
    return data.candidates[0].content.parts[0].text;
}
```

**构建 messages 数组的结构：**

```javascript
buildMessages(userMessage, conversationHistory, gameState) {
    const messages = [];

    // 1. System Prompt（世界观 + NPC人设 + 格式规范）
    messages.push({
        role: 'system',
        content: this.buildSystemPrompt()
    });

    // 2. 当前游戏状态快照
    messages.push({
        role: 'system',
        content: `【当前游戏状态】\n${this.buildStateContext(gameState)}`
    });

    // 3. 对话历史（取最近 N 条）
    const recentHistory = conversationHistory.slice(-20);
    messages.push(...recentHistory);

    // 4. 本次用户输入
    messages.push({
        role: 'user',
        content: userMessage
    });

    return messages;
}
```

### 3.4 改造 `handleMessageOutput()` — 核心改造点

**文件：`module/game-helpers.js`**

这是最关键的改造。当前此函数将消息注入 ST 聊天并触发 AI 生成。改造后需要：

1. 将用户消息存入本地聊天历史
2. 构建完整 prompt
3. 直接调用 LLM API
4. 解析 AI 回复
5. 更新游戏状态
6. 在 UI 中展示结果

**改造后的逻辑流程：**

```javascript
async function handleMessageOutput(message) {
    const unprocessed_message = message;

    // 1. 消息预处理（移除属性变化/计算过程 —— 保留现有逻辑）
    message = removeMetaInfo(message);

    // 2. 检测新周
    detectNewWeek(message);

    // 3. 保存为上一条用户消息
    lastUserMessage = message;

    // 4. 在 UI 中显示"用户消息"
    chatPanel.addMessage('user', unprocessed_message);

    // 5. 添加到对话历史
    apiService.conversationHistory.push({ role: 'user', content: message });

    // 6. 构建完整的 messages 数组
    const gameState = getCurrentGameState();
    const messages = promptBuilder.buildMessages(message, 
        apiService.conversationHistory, gameState);

    // 7. 显示加载状态
    chatPanel.showLoading();

    try {
        // 8. 调用 LLM API
        const aiResponse = await apiService.sendMessage(messages);

        // 9. 在 UI 中显示 AI 回复
        chatPanel.addMessage('assistant', aiResponse);

        // 10. 解析 AI 回复（复用现有 parseOutput 逻辑）
        parseOutput(aiResponse);

        // 11. 添加到对话历史
        apiService.conversationHistory.push({ role: 'assistant', content: aiResponse });

        // 12. 保存游戏状态
        await storageService.saveGameState(gameData);
        await storageService.saveConversationHistory(apiService.conversationHistory);

    } catch (error) {
        console.error('AI 调用失败:', error);
        chatPanel.showError(error.message);
    } finally {
        chatPanel.hideLoading();
    }
}
```

```javascript
async function handleRegenerate() {
    // 移除最后一条 AI 回复
    if (apiService.conversationHistory.length > 0) {
        const last = apiService.conversationHistory[apiService.conversationHistory.length - 1];
        if (last.role === 'assistant') {
            apiService.conversationHistory.pop();
        }
    }
    chatPanel.removeLastMessage();

    // 用最后一条用户消息重新触发生成
    const lastUserMsg = apiService.conversationHistory
        .filter(m => m.role === 'user').pop();
    if (lastUserMsg) {
        // 重新构建并发送
        const gameState = getCurrentGameState();
        const messages = promptBuilder.buildMessages(
            lastUserMsg.content, apiService.conversationHistory, gameState);
        chatPanel.showLoading();
        try {
            const aiResponse = await apiService.sendMessage(messages);
            chatPanel.addMessage('assistant', aiResponse);
            parseOutput(aiResponse);
            apiService.conversationHistory.push({ role: 'assistant', content: aiResponse });
            await storageService.saveGameState(gameData);
        } catch (error) {
            chatPanel.showError(error.message);
        } finally {
            chatPanel.hideLoading();
        }
    }
}
```

### 3.8 新建聊天面板

**文件：`ui/chat-panel.js`**

当前项目中，AI 的回复是在 ST 的聊天窗口中显示的，游戏本身只通过 `parseOutput()` 提取数据。独立化后需要：

**方案选择（二选一）：**

**方案 A：纯游戏模式（推荐）**
- 不显示完整的 AI 原文，只显示 `parseOutput()` 提取到的"正文"部分
- 正文展示在现有的 `story-text` 区域（已有的 SLG 故事面板）
- 优点：UI 不需要大改，保持当前游戏风格
- 缺点：调试困难，看不到 AI 完整回复

**方案 B：聊天 + 游戏混合模式**
- 新增一个可展开/收起的聊天面板，显示完整的 AI 对话
- 游戏界面保持不变，聊天面板作为侧边栏或底部抽屉
- 优点：可调试，用户能看到完整交互
- 缺点：UI 改动较大

**建议：先实现方案 A，再可选开启方案 B 的调试模式。**

```javascript
class ChatPanel {
```

### 3.9 改造 `game-utils.js` — 环境检测

**当前的三个检测函数需要改造：**

```javascript
// 改造后：始终返回独立模式的标识
function isInRenderEnvironment() {
    // 保留检测逻辑，兼容可能的 ST 嵌入场景
    if (typeof STscript === 'function' || typeof triggerSlash === 'function') {
        return true;  // 仍在 ST 中运行
    }
    return typeof window.apiService !== 'undefined';  // 独立模式也返回 true
}

    if (typeof triggerSlash === 'function') return 'otherRenderer';
```

## 五、需要注意的技术问题

### 5.1 CORS 跨域

直接从浏览器调用 LLM API 可能遇到 CORS 问题：

- **OpenAI 官方 API**：不允许浏览器直接调用（需后端代理）
- **第三方中转站**：通常已配置 CORS 允许
- **Gemini API**：允许浏览器直接调用

**解决方案：**
- 推荐用户使用支持 CORS 的中转站/代理
- 或者提供一个极简的本地代理脚本（Node.js/Python）
- 或使用 Cloudflare Workers 做 CORS 代理

### 5.2 API 密钥安全

浏览器端存储 API 密钥存在安全风险：
- localStorage 中的密钥可被同域 XSS 攻击读取
- 用户需要了解风险

**缓解措施：**
- 在配置界面添加安全提示
- 密钥输入框使用 `type="password"`
- 不在 console.log 中输出密钥

### 5.3 对话历史的 Token 管理

独立前端需要自行管理上下文窗口：
- 保留最近 N 条对话（建议 10-20 条）
- system prompt + 历史 + 当前输入的总 Token 不超过模型上限
- 可参考 cs2 的 `conversationHistory.slice(-10)` 策略
- 后续可实现"总结压缩"策略（让 AI 总结旧对话）

### 5.4 `parseOutput()` 的兼容性

当前 `parseOutput()` 假设 AI 回复中包含 `<SUMMARY>` 标签和 JSON 数据。独立化后：
- AI 回复格式不变（由提示词控制）
- `removeThinkingContent()` 仍然需要（过滤 thinking 标签）
- 确保提示词中的格式规范在独立版中也能正确加载

### 5.5 静态资源部署

独立前端需要一个 HTTP 服务器（`file://` 协议下 `fetch` 不可用）：
- 开发时：`npx serve .` 或 VS Code Live Server
- 部署时：GitHub Pages / Vercel / Netlify / 任意静态托管

### 5.6 双模式兼容

建议保留 ST 兼容性，使代码支持两种运行模式：

```javascript
const isStandalone = !getRenderFunction();

// 在关键调用点进行分支
if (isStandalone) {
    // 独立模式逻辑
} else {
    // ST 模式（保留原逻辑）
}
```

这样用户既能在 ST 中使用，也能独立运行。

## 八、开发级设计方案

这一章是在前文改造方向基础上的落地设计，目标不是再讨论“要不要做”，而是明确“代码应如何组织、模块如何协作、数据如何落盘、运行时如何流转”。

### 8.1 模块划分

推荐在保持现有全局函数式结构的前提下，新增一层“服务模块”，不要直接把所有新逻辑继续堆回 [index.html](e:/JJBurst/git/index.html)。

建议结构：

```
api/
  api-service.js             # 主 LLM 对话调用
  embedding-service.js       # 向量化服务，支持三种方案与自动降级
  prompt-builder.js          # Prompt 组装
  response-parser.js         # 对 parseOutput 的模块化封装

storage/
  storage-service.js         # IndexedDB 基础存储
  memory-service.js          # 对话历史、摘要记忆、召回逻辑
  save-service.js            # 存档/读档与快照

runtime/
  app-runtime.js             # 页面启动、依赖初始化、运行时协调
  pipeline.js                # 单轮消息处理流水线

ui/
  config-modal.js            # API 与 embedding 配置
  save-manager.js            # 存档列表与读档界面
  chat-panel.js              # 可选调试聊天面板

module/
  game-state.js              # 继续负责变量同步，但持久化委托给 storage-service
  game-helpers.js            # 继续保留 UI/交互辅助，消息发送改为委托 pipeline
```

各模块职责建议如下：

| 模块 | 核心职责 | 不负责的内容 |
|------|------|------|
| `APIService` | 发送聊天请求、处理模型返回、封装 OpenAI/Gemini/Claude 差异 | 不负责 prompt 拼装、不负责持久化 |
| `EmbeddingService` | 提供 `Qwen embedding -> Transformers.js -> 关键词权重` 的统一向量化接口 | 不负责历史筛选、不负责 UI |
| `PromptBuilder` | 读取提示词文件、构建 system prompt、拼装当前状态和历史上下文 | 不直接调用 fetch API |
| `ResponseParser` | 把 AI 回复拆成 `mainText`、`summary`、`sideNote`、`raw` | 不直接修改 DOM |
| `MemoryService` | 保存最近原文历史、摘要记忆入库、向量召回、标签过滤 | 不负责游戏数值更新 |
| `StorageService` | 提供 IndexedDB 的读写抽象和事务 | 不负责任何业务规则 |
| `SaveService` | 存档、读档、快照恢复 | 不负责 embedding |
| `AppRuntime` | 初始化顺序、模块依赖注入、页面启动 | 不承载业务细节 |
| `Pipeline` | 协调“发消息 -> 解析 -> 更新状态 -> 存储 -> 向量化 -> UI” | 不直接持有状态数据 |

#### 8.2.2 EmbeddingService

这个模块是向量化能力的核心，必须把三种方案统一在一个接口下面。

```javascript
class EmbeddingService {
    constructor(configStore) {
        this.configStore = configStore;
        this.activeMethod = 'qwen';
    }

    async init() {}
    getActiveMethod() {}
    async healthCheck() {}
    async embedText(text, options = {}) {}
    async embedBatch(texts, options = {}) {}
    async fallbackEmbed(text, options = {}) {}
}
```

`embedText` 的统一返回结构建议为：

```javascript
{
    ok: true,
    method: 'qwen', // 'qwen' | 'transformers' | 'keyword'
    vectorType: 'dense', // 'dense' | 'sparse'
    vector: [],
    dimensions: 1024,
    provider: 'siliconflow',
    model: 'Qwen/Qwen3-Embedding-4B',
    degradedFrom: null
}
```

三种实现建议：

1. `embedWithQwen(text)`
作用：调用硅基流动 embedding 接口。

2. `embedWithTransformers(text)`
作用：浏览器加载本地模型生成 dense 向量。

3. `embedWithKeywords(text)`
作用：做关键词提取、n-gram 拆分和权重累加，输出 sparse 向量对象。

降级顺序固定：

```javascript
Qwen embedding -> Transformers.js -> 关键词权重 -> 返回 null
```

### 8.4 运行时调用顺序

这部分建议明确成流水线，否则后面很容易在 [module/game-helpers.js](e:/JJBurst/git/module/game-helpers.js) 和 [module/game-state.js](e:/JJBurst/git/module/game-state.js) 之间出现职责混乱。

#### 8.4.1 页面启动流程

```javascript
window.onload
  -> AppRuntime.init()
     -> StorageService.init()
     -> APIService.init()
     -> EmbeddingService.init()
     -> MemoryService.init()
     -> PromptBuilder.init()
     -> loadOrInitGameData()
     -> restoreRecentConversation()
     -> updateAllDisplays()
```

启动顺序要求：

1. 先初始化存储层。
2. 再初始化 API 和 embedding。
3. 再初始化依赖它们的 `MemoryService` 和 `PromptBuilder`。
4. 最后再恢复 `gameData` 和近期历史。

#### 8.4.2 单轮消息处理流程

这个流程是整个独立前端最重要的主链路。

```javascript
performAction/sendFreeAction/sendInteraction
  -> handleMessageOutput(message)
  -> Pipeline.runTurn(message)
     -> preprocessMessage(message)
     -> appendRecentConversation(userMessage)
     -> PromptBuilder.buildMessages(...)
     -> APIService.sendMessages(messages)
     -> ResponseParser.parse(aiResponse)
     -> applySideNoteToGameState(parsed.sideNote)
     -> renderMainText(parsed.mainText)
     -> appendRecentConversation(assistantMessage)
     -> buildMemoryRecord(parsed, context)
     -> saveMemoryRecord(record)
     -> async embedAndPersist(record)
     -> saveAppState()
```

这里要特别注意两点：

1. `SIDE_NOTE` 应先更新状态，再保存状态快照。
2. 向量化可以异步，但 `SUMMARY` 和 `memoryRecord` 的元数据应同步入库，否则失败后无法补建索引。

#### 8.4.3 Prompt 构建流程

```javascript
PromptBuilder.buildMessages({ userMessage, gameData })
  -> buildSystemPrompt()
  -> buildStateContext(gameData)
  -> MemoryService.getRecentConversation(8)
  -> MemoryService.retrieveMemories(query, { topK: 3, minTurnGap: 4 })
  -> formatRetrievedMemories()
  -> merge messages
```

注入顺序建议严格保持：

1. system prompt
2. 当前状态快照
3. 远期摘要召回
4. 最近窗口原文
5. 当前用户输入

不要把远期摘要放在最近窗口后面，否则最新上下文容易被远期记忆稀释。

#### 8.4.4 向量化与降级流程

```javascript
EmbeddingService.embedText(queryText)
  -> try embedWithQwen()
  -> if failed: try embedWithTransformers()
  -> if failed: try embedWithKeywords()
  -> if failed: return { ok: false }
```

降级触发条件建议：

1. Qwen 失败：网络错误、401、429、超时。
2. Transformers.js 失败：模型加载失败、浏览器不支持、用户关闭本地模型。
3. 关键词方案失败：通常只会是实现错误，此时直接关闭远期记忆召回。

#### 8.4.5 召回流程

```javascript
MemoryService.retrieveMemories(query, options)
  -> buildRetrievalQuery(currentLocation, currentNpc, userMessage, lastSummary)
  -> filter memory_records by gameId
  -> exclude recent turnIds
  -> optional filter by location/eventType/npcIds
  -> load corresponding vectors
  -> compute similarity
  -> rerank by tags
  -> take topK
  -> format as summary lines
```

最终返回给 `PromptBuilder` 的文本格式建议为：

```text
【远期相关记忆】
- 第1年第3月第2周，在演武场，你与萧白瑚切磋后关系缓和。
- 第1年第4月第1周，在藏经阁，你从姬姒处得知门派旧事。
- 第1年第4月第3周，在后山，你遭遇战斗事件并击退蒙面人。
```

#### 8.4.6 存档与读档流程

```javascript
saveGame()
  -> syncGameDataFromVariables()
  -> StorageService.getAppState()
  -> MemoryService.getRecentConversation()
  -> SaveService.createSaveSlot()

loadGame(saveId)
  -> SaveService.loadSaveSlot(saveId)
  -> restore gameData
  -> restore recentConversation
  -> refresh UI
```

建议：

1. 存档只保存当前状态和最近窗口，不必把整个向量库做多份快照。
2. 长期记忆库继续按 `gameId` 常驻，不跟着每个存档复制。
3. 如果用户希望“历史记忆也随存档完全回滚”，那是下一阶段功能，不建议一开始就做。

#### 8.4.7 重新生成流程

```javascript
handleRegenerate()
  -> remove last assistant message from recentConversation
  -> rollback gameData to previous snapshot
  -> rebuild prompt with last user message
  -> resend
  -> replace recentConversation tail
  -> overwrite current appState
```

这里建议额外新增 `turn_snapshots` 或在 `conversation_history` 中保存每轮结束时的 `gameDataDigest`，否则重新生成时难以安全回滚。

### 8.5 推荐的最小落地顺序

如果按开发顺序推进，建议按下面的依赖关系实施：

1. 先写 `StorageService` 和 IndexedDB 表结构。
2. 再写 `APIService`、`ResponseParser`。
3. 再把 [module/game-helpers.js](e:/JJBurst/git/module/game-helpers.js) 中的 `handleMessageOutput` 改为调用 `Pipeline.runTurn()`。
4. 然后实现 `MemoryService` 的“最近窗口 + `<SUMMARY>` 入库”，先不做 embedding。
5. 再接 `EmbeddingService` 的关键词权重版本。
6. 再接 Qwen embedding。
7. 最后接 Transformers.js 降级方案和调试面板。

这样可以保证每一步都能运行，不会出现“必须一次性改完整套系统才能启动”的风险。

# 悬赏玩法设计文档

## 一、功能概述

在议事厅新增"悬赏任务"入口，玩家可通过 LLM 生成一批悬赏通缉令，接取后在大地图对应地点触发定向战斗，根据战斗结果获得声望和赏金奖励。

**仅影响独立前端链路（index.html），不影响 index - SR.html 的 SillyTavern 链路。**

---

## 二、完整事件流（两轮 LLM 交互）

```
[议事厅] 点击"悬赏任务"按钮
    ↓
显示悬赏弹窗
  ├─ 无任务时：弹窗内容为空，底部显示"发布悬赏"按钮
  └─ 有进行中任务时：高亮显示当前接取的任务卡，底部显示"放弃任务"按钮
    ↓
点击"发布悬赏"
    ↓
调用 generateBountyTasks()
    → LLM API 请求（参考 summary-runner.js 的调用方式）
    → 解析 JSON 响应，得到 3 个悬赏任务
    → 以任务卡形式渲染到弹窗
    ↓
玩家点击某张任务卡（接取任务）
    → acceptBounty(task) 将任务写入 activeBounty（同步到 gameData）
    → 弹窗关闭
    ↓
玩家前往 [山门] → 点击"下山"
    → showWorldMap() 附带 bountyLocation 参数传给大地图 iframe
    ↓
[world_map.html]
    → 识别 bountyLocation 参数
    → 在对应地点图标上叠加"⚔"悬赏标记
    → 玩家选择该地点出发时，确认弹窗显示"⚠ 此地有悬赏目标！"
    → confirmTravel() 的 postMessage 中添加 bountyBattle: 1
    ↓
[game-events.js] worldmap-exit 消息处理
    → 检测到 bountyBattle === 1
    → 构造 user 输入：
        时间/季节/地点/随行NPC
        ${_actorName}行动选择：悬赏缉拿
        （不在 user message 中重复悬赏目标，行动指导 EJS 模板已通过 gameData.activeBounty 包含完整信息）
    → 设置 currentBattleType = 'bounty'（在发送前设置，供后续判断用）
    → 调用 handleMessageOutput(userMessage)
    ↓
════════════════════════════════
 第一轮 LLM 交互
════════════════════════════════
    → prompt-builder.js 识别 key "行动选择：悬赏缉拿"
    → worldbook-engine 自动注入 PROMPT_ACTION_BOUNTY 行动指导
      （指导 LLM：叙述下山追踪的过程，在 SIDENOTE 中构造悬赏战斗事件）
    → LLM 输出：
        MAIN_TEXT：下山→路途→抵达目的地→遭遇悬赏目标的剧情叙述
        SIDENOTE：包含 随机事件（事件类型：战斗事件），enemy 信息与 activeBounty 一致
    ↓
parseLLMResponse 解析第一轮响应
    → 处理 MAIN_TEXT（正常显示剧情）
    → 检测到 随机事件.事件类型 === '战斗事件'
    → 若 currentBattleType === 'bounty'：
        用 activeBounty 覆盖 currentBattleEvent（忽略 SIDENOTE 中的名称/武学/报酬，以变量为准）
    → displayBattleEvent(currentBattleEvent)
    → 显示"开始战斗"按钮，等待玩家点击
    ↓
玩家点击"开始战斗"
    → 在按钮 handler 中保护 currentBattleType：
        if (currentBattleType !== 'bounty') currentBattleType = 'event'
        ← 只防止被意外覆盖，不重新推导（worldmap-exit 已赋值为准）
    → showBattleGame(battleData) 开启战斗 iframe
    ↓
战斗结束，[game-events.js] battle-exit 消息处理
    → currentBattleType === 'bounty' 分支
    ↓
【变量结算阶段（在发请求前完成，确保快照干净）】
    ① 同步道具数量（remainingItems）
    ② 胜利时：应用声望+赏金奖励，generateBattleEventDrop() 掉落计算
    ③ 清空 activeBounty = null
    ④ syncGameDataFromVariables()  ← 把①②③全部写回 gameData
    ⑤ showModal() 显示奖励弹窗（异步，不阻塞后续）
    ↓
【发起第二轮请求】
    ⑥ handleMessageOutput(battleResultMessage)
        → 内部自动 saveFullSnapshot()  ← 快照保存已结算、已清空任务的干净状态
        → 内部自动 saveAppState()
        → pipeline.runTurn()
    ↓
════════════════════════════════
 第二轮 LLM 交互
════════════════════════════════
    → LLM 正常输出战斗后的后续剧情
    ↓
parseLLMResponse 解析第二轮响应（普通流程）

【重生成安全性】
    若第二轮请求失败重生成，restoreFromSnapshot() 恢复的是步骤④之后的状态
    奖励已结算、activeBounty 已清空，不会发生重复结算
```

## 四、LLM Prompt 设计

### 4.1 System Prompt

```
你是武侠世界的悬赏公告栏。请生成3张不同的悬赏通缉令。要求：
- 风格贴近五代十国末期西域武侠风格
- 人名/绰号有江湖气息
- 罪行描述简洁有画面感，不超过50字
- 地点必须从给定的列表中选择
- 难度分布合理，3个任务等级不完全相同
严格按 JSON 格式输出，不输出任何其他文字。
```

### 4.2 User Prompt（动态构造）

```
当前游戏背景：北宋初期，西域天山派。
玩家当前武学值：{playerStats.武学}（0-100）。
本周：第{year}年第{month}月第{week}周，季节：{season}。

可选地点列表：高昌、沙州、龟兹、伊州、哈密绿洲、月牙泉
（地点名称必须与 world_map.html 的地点 key 完全一致，如实际 key 为 '沙州' 而非 '沙洲城'）

请生成3张悬赏通缉令，JSON格式如下（直接输出JSON，不加任何markdown代码块）：
{"bounties":[
  {"enemyName":"...","locationName":"高昌","level":3,"description":"...","reputationReward":15,"goldReward":300},
  {"enemyName":"...","locationName":"沙州","level":5,"description":"...","reputationReward":25,"goldReward":500},
  {"enemyName":"...","locationName":"哈密绿洲","level":7,"description":"...","reputationReward":35,"goldReward":700}
]}

level建议参考玩家武学值：武学<20对应1-3级，武学20-50对应2-5级，武学50-80对应3-7级，武学>80对应5-9级。
reputationReward约为level×5，goldReward约为level×100。
```

### 4.3 响应解析

```javascript
// 提取 JSON 字符串（容错 markdown 代码块包裹的情况）
function parseBountyResponse(text) {
    const cleaned = text.replace(/```json|```/g, '').trim();
    const data = JSON.parse(cleaned);
    if (!Array.isArray(data.bounties) || data.bounties.length === 0) throw new Error('格式无效');
    return data.bounties;
}
```

## 六、设计决策确认

### Q1：任务时效性
**决策：暂不加。** `acceptedWeek` 字段从 activeBounty 中移除，保持结构简洁。

### Q2：弹窗展示逻辑
**决策：有已接任务时，弹窗仅显示该任务卡（特殊高亮标记），底部只有"放弃任务"和"关闭"按钮，"发布悬赏"按钮不显示。放弃任务确认后，弹窗停留并切换回无任务状态（显示空列表+"发布悬赏"按钮），方便直接重新生成。**

弹窗的两种状态：
```
【无任务状态】              【有任务状态】
┌─────────────┐            ┌─────────────┐
│  悬赏公告    │            │  悬赏公告    │
│  （空）      │            │ ⚔ [当前任务] │  ← 高亮显示，标注"进行中"
│             │            │  目标/地点/  │
│             │            │  等级/奖励   │
├─────────────┤            ├─────────────┤
│ [发布悬赏]  │            │ [放弃任务]  │  ← 确认后切换回左侧状态
│ [关闭]      │            │ [关闭]      │
└─────────────┘            └─────────────┘
```

### Q3：地点名称

### Q4：悬赏战斗背景图
**决策：复用 `showBattleGame()` 中已有的 `window.__lastValidSceneUrl` 逻辑。**

由于悬赏缉拿的第一轮 LLM 输出会更新 `__lastValidSceneUrl`（正文中含目的地场景描写），`showBattleGame()` 被触发时 `GameMode === 1`，会自动取到最新的场景图，无需额外处理。

### Q5：重复接取限制
**决策：暂不限制。** 目标和地点每次由 LLM 随机生成，重复概率低；战斗结束后变量自动清空，机制上已有天然约束。

### Q6：放弃任务确认弹窗
**决策：加确认弹窗。** 点击"放弃任务"后弹出确认，确认后清空 `activeBounty`，弹窗切换回无任务状态而非关闭。

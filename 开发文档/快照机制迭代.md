# 快照机制迭代方案

## 1. 背景与目标

### 现状问题

当前快照（`snapshot`）仅保存发送前的 `gameData`，其余字段（`uiConversation`、`summaryHistory`、`weekHistory`、`markWeekUiIndex`、`summaryBuff`）在出错回退或重新生成时，各自有一套独立的回退逻辑，导致：

- `_confirmRegenerate()` 中存在基于 `lastTurnCommitted` 标志位的多层分支（新周/普通轮、runTurn初版/runSummary最终版）
- `pipeline.js` 的 commit catch 中手动维护 `uiConversationBeforeCommit` / `summaryHistoryBeforeCommit` 两份备份
- embedding 需要单独执行 `deleteEmbedding` + `removeFromCache`
- 新增字段时必须同步补写回退逻辑，极易遗漏

### 迭代目标

将快照升级为**全量快照**，覆盖所有需要回退的字段。回退逻辑退化为单一操作：**从 `snapshot_db` 整体还原**，彻底消除分支判断与字段遗漏风险。

---

## 2. 新架构概述

新增一个独立 IndexedDB 数据库 `jxz_snapshot_db`，与现有 `jxz_db` 平级。`jxz_snapshot_db` 的 keyval store 结构与 `jxz_db` 完全相同，保存发送前所有需要回退的字段的完整状态。

```
IndexedDB
├── jxz_db（主库，已有）
│   ├── appState
│   ├── uiConversation
│   ├── summaryHistory
│   ├── weekHistory
│   ├── markWeekUiIndex
│   ├── summaryBuff
│   ├── saveIndex
│   ├── save_xxxx（存档槽）
│   └── emb_xxxx（embedding 向量）
│
└── jxz_snapshot_db（新增，快照库）
    ├── appState          ← gameData 深拷贝
    ├── uiConversation    ← 发送前的 UI 对话历史
    ├── summaryHistory    ← 发送前的摘要历史
    ├── weekHistory       ← 发送前的周历史
    ├── markWeekUiIndex   ← 发送前的周定位索引
    ├── summaryBuff       ← 发送前的周总结缓冲
    └── lastUserMessage   ← 本次用户输入内容（字符串）
```

### snapshot_db 明确排除的字段

| 字段 | 排除原因 |
|---|---|
| `save_xxxx` / `saveIndex` | 手动存档槽，与当前对话轮次无关，回退不应影响存档 |
| `emb_xxxx` | 体积大（每条 ~6KB），且 `_syncEmbeddingsWithSummaryHistory()` 在每轮 `runTurn()` 开始时会自动按 summaryHistory 修正孤立条目，无需快照 |
| `lastTurnCommitted` | 该字段废弃（详见第 6 节） |

## 3. 数据库实现方案

### 3.1 新增 `idb-snapshot.js`（对应 `idb-storage.js`）

新增一个与 `idb-storage.js` 结构完全相同的模块，仅修改数据库名：

```javascript
var DB_NAME = 'jxz_snapshot_db';
var DB_VERSION = 1;
var STORE_NAME = 'keyval';
```

对外暴露 `idbSnapshot` 对象，接口与 `idbStorage` 相同（`open`, `put`, `get`, `remove`, `getAll`）。

### 3.2 `storage-service.js` 中新增快照 API

在 `storageService` 中新增以下一组 API，专门操作 `jxz_snapshot_db`：

```javascript
// 写入全量快照（发送前调用）
async function saveFullSnapshot()

// 从全量快照还原（出错/重生成时调用）
async function restoreFromSnapshot()

// 快照是否存在
function hasSnapshot()

// 清空快照（读档/导入存档后调用）
function clearSnapshot()

// 读取快照中的 lastUserMessage（重生成弹窗用）
function getSnapshotLastUserMessage()

// 更新快照中的 lastUserMessage（重生成确认编辑后用）
function updateSnapshotLastUserMessage(msg)
```

#### `saveFullSnapshot()` 的实现逻辑

```javascript
async function saveFullSnapshot() {
    var snap = {
        appState:        { gameData: structuredClone(gameData) },
        uiConversation:  structuredClone(loadUIConversation()),
        summaryHistory:  structuredClone(summaryHistoryService.getAll()),
        weekHistory:     structuredClone(weekHistoryService.getAll()),
        markWeekUiIndex: getMarkWeekUiIndex(),
        summaryBuff:     structuredClone(getSummaryBuff()),
        lastUserMessage: (typeof lastUserMessage !== 'undefined') ? lastUserMessage : ''
    };
    // 写入内存缓存
    _snapshotCache = snap;
    // 逐字段写入 jxz_snapshot_db（异步，不阻塞发送）
    await Promise.all([
        idbSnapshot.put('appState',        snap.appState),
        idbSnapshot.put('uiConversation',  snap.uiConversation),
        idbSnapshot.put('summaryHistory',  snap.summaryHistory),
        idbSnapshot.put('weekHistory',     snap.weekHistory),
        idbSnapshot.put('markWeekUiIndex', snap.markWeekUiIndex),
        idbSnapshot.put('summaryBuff',     snap.summaryBuff),
        idbSnapshot.put('lastUserMessage', snap.lastUserMessage),
    ]);
    // localStorage 降级：仅存 appState（体积可控）
    _lsSet(LS_SNAPSHOT, snap.appState);
}
```

#### `restoreFromSnapshot()` 的实现逻辑

```javascript
async function restoreFromSnapshot() {
    var snap = _snapshotCache;
    if (!snap) return false;

    // 还原 gameData
    gameData = mergeWithDefaults(snap.appState.gameData, defaultGameData);
    syncVariablesFromGameData();
    saveAppState({ gameData: gameData });

    // 还原 uiConversation
    replaceUIConversation(snap.uiConversation);

    // 还原 summaryHistory
    summaryHistoryService.importAll(snap.summaryHistory);

    // 还原 weekHistory
    weekHistoryService.importAll(snap.weekHistory);

    // 还原 markWeekUiIndex
    setMarkWeekUiIndex(snap.markWeekUiIndex);

    // 还原 summaryBuff
    if (snap.summaryBuff) {
        setSummaryBuff(snap.summaryBuff);
    } else {
        clearSummaryBuff();
    }

    return true;
}
```

## 4. 快照写入时机

**只有一个写入时机**：用户点击发送、API 请求发起前。

在 `game-helpers.js` 的发送函数（当前约第 160 行）中：

```javascript
// 现有代码（改造前）：
storageService.setLastTurnCommitted(false);
syncGameDataFromVariables();
storageService.saveSnapshot(structuredClone(gameData));
storageService.saveAppState({ gameData: gameData });
// ... 500ms setTimeout 后调用 pipeline.runTurn()

// 改造后：
syncGameDataFromVariables();
await storageService.saveFullSnapshot();   // ← 全量快照写入完成后才继续
storageService.saveAppState({ gameData: gameData });
await pipeline.runTurn(message);           // ← 快照落盘后再发起请求，去掉 setTimeout
```

> **关于 setTimeout 缓冲**：现有代码用 500ms `setTimeout` 做写入缓冲，这是不安全的——数据量大时写入可能超时，快照未落盘就已发出请求，此时若 AI 出错则快照回退不可靠。改为 `await saveFullSnapshot()` 后，快照写入是强前置条件，彻底消除这一风险。发送函数本身需确保是 `async` 函数（当前已是）。

### 快照不在以下时机写入

| 时机 | 原因 |
|---|---|
| 重新生成时 | 只读取快照，不写入，保证多次重生成回到同一状态点 |
| 读档/导入存档时 | 需清空快照（见第 7 节） |
| 自动存档时 | 存档记录的是提交后状态，与快照无关 |
| pipeline commit 阶段 | 只更新主库 |

## 5. 重新生成机制（简化后）

替换现有 `_confirmRegenerate()` 中约 80 行的分支逻辑，统一为：

```javascript
async function _confirmRegenerate() {
    var editedMsg = /* 从弹窗获取 */;
    closeModal();
    if (!editedMsg) { showModal('用户消息不能为空'); return; }
    if (!storageService.hasSnapshot()) { showModal('没有可重新生成的操作'); return; }

    // 1. 更新快照中的 lastUserMessage（用户可能编辑了消息）
    storageService.updateSnapshotLastUserMessage(editedMsg);
    lastUserMessage = editedMsg;

    // 2. 从快照还原所有字段
    var ok = await storageService.restoreFromSnapshot();
    if (!ok) { showModal('快照数据损坏，无法重新生成'); return; }

    // 3. 刷新 UI 显示
    if (typeof updateAllDisplays === 'function') updateAllDisplays();
    if (typeof updateSceneBackgrounds === 'function') updateSceneBackgrounds();
    // ... 其他 UI 刷新

    // 4. 重发请求
    pipeline.runTurn(editedMsg, { isRegenerate: true }).catch(function(e) {
        console.error('[Regenerate] 重新生成失败:', e);
    });
}
```

**原逻辑中可完全删除的代码块：**
- `didLastTurnCommit()` 判断
- `_popFromWeek` / 新周子场景分支
- `summaryHistoryService.popLast()` / `weekHistoryService.popLast()`
- `storageService.popLastAssistantUI()`
- `storageService.deleteEmbedding()` / `memoryRecall.removeFromCache()`
- `storageService.clearSummaryBuff()`
- `summaryBuff.prevMarkWeekUiIndex` 的恢复逻辑

## 6. AI 出错回退机制（简化后）

替换现有 `pipeline.js` 中 `_commitResponse()` 的 catch 块：

```javascript
// 现有（改造前）：
} catch (err) {
```

## 7. 存档机制（自动存档/读档/导出/导入）

### 7.1 自动存档（`createSave`）

快照不写入存档 payload，同时从 payload 结构中**移除** `snapshot` 和 `lastTurnCommitted` 字段：

```javascript
// 移除这两行：
snapshot: structuredClone(loadSnapshot()),
lastTurnCommitted: didLastTurnCommit(),
```

### 7.2 读档（`loadSave` + `index.html` 的加载逻辑）

加载存档后，**清空 snapshot_db**，并删除以下代码：

```javascript
// 删除：
storageService.setLastTurnCommitted(...);
// 删除：
if (payload.snapshot) {
    storageService.saveSnapshot(payload.snapshot);
} else {
    storageService.clearSnapshot();
}
```

改为：

```javascript
storageService.clearSnapshot();  // 读档后无快照，需发送一次才能重生成
```

**副作用（已知且合理）**：加载存档后立即点击"重新生成"会提示"没有可重新生成的操作"，玩家需发送一次消息后方可重生成。这比旧逻辑中"可能使用过期快照"更安全。

#### 重生成按钮的禁用状态

以下三种场景完成后，`snapshot_db` 不存在或已清空，**前端重生成按钮应立即置为禁用**：

- 开始新游戏
- 读档（加载存档槽）
- 导入 JSON 存档文件

需要新增一个专门负责刷新重生成按钮可用性的函数，并在上述三个场景结束时调用：

```javascript
function updateRegenerateButtonState() {
    var btn = document.getElementById('regenerate-btn');
    if (!btn) return;
    var hasSnap = (typeof storageService !== 'undefined') && storageService.hasSnapshot();
    // 无快照时直接禁用，不再需要额外判断 lastTurnCommitted
    if (!hasSnap) {
        btn.disabled = true;
        btn.style.opacity = '0.5';
        btn.style.cursor = 'not-allowed';
        btn.title = '发送一次消息后方可重新生成';
        return;
    }
    // 有快照时，保留原有的特殊事件屏蔽逻辑
    // ...（原 _blockRegen 逻辑不变）
}
```

同时，`updateFreeActionInputState()` 中现有的 `regenerateBtn` 判断逻辑需同步引入 `hasSnapshot()` 检查：
- `inputEnable=0` 分支：无快照时同样禁用（现在已禁用，不变）
- `inputEnable=1` 分支：在原有的 `_blockRegen` 计算基础上，增加 `!storageService.hasSnapshot()` 作为额外禁用条件

原来用 `didLastTurnCommit()` 判断是否禁用的逻辑（`_committed` 变量）可以**一并删除**，因为新方案不再依赖该标志位。

### 7.3 导出 JSON（`buildSavePayload` / `exportSaveToJson`）

同上，从 `buildSavePayload()` 返回值中移除 `snapshot` 和 `lastTurnCommitted` 字段。导出文件不包含快照数据（快照是"飞行中状态"，不属于存档内容）。

### 7.4 导入 JSON（`importSavePayload` / `importSaveFromJson`）

移除导入后恢复 `lastTurnCommitted` 和 `snapshot` 的代码，改为调用 `storageService.clearSnapshot()`。

## 9. localStorage 兼容方案

### 设计原则

localStorage 有容量限制（通常 5MB），snapshot_db 包含 `uiConversation`、`summaryHistory` 等大体积字段，**无法全量写入 localStorage**。

| 字段 | localStorage 策略 |
|---|---|
| `appState`（gameData） | 写入 `jxz_snapshot`（与现在相同，体积可控） |
| `uiConversation` | **仅 IDB**，LS 不存 |
| `summaryHistory` | **仅 IDB**，LS 不存 |
| `weekHistory` | **仅 IDB**，LS 不存 |
| `markWeekUiIndex` | **仅 IDB**，LS 不存（数字，可以单独存但意义不大） |
| `summaryBuff` | **仅 IDB**，LS 不存 |
| `lastUserMessage` | 写入 `jxz_snapshot_lastMsg`（字符串，极小） |

### 降级行为（IDB 不可用时）

当 IDB 不可用，从 localStorage 加载快照时：
- `gameData` 可以还原（从 `jxz_snapshot`）
- `uiConversation`、`summaryHistory` 等**无法还原**，快照视为无效，提示"没有可重新生成的操作"

这是可接受的降级：IDB 不可用时整体功能已大幅降级，快照只是锦上添花。

## 10. 初始化与迁移

### 启动初始化（`storageService.init()`）

新增 `jxz_snapshot_db` 的初始化：
```javascript
await idbSnapshot.open();
var snapshotData = await idbSnapshot.getAll();
if (Object.keys(snapshotData).length > 0) {
    _snapshotCache = snapshotData;
} else {
    // 尝试从旧 jxz_snapshot（LS）迁移 appState 部分
    var oldSnap = _lsGet('jxz_snapshot');
    if (oldSnap) {
        _snapshotCache = { appState: oldSnap };
        idbSnapshot.put('appState', oldSnap);
        // 迁移完成后清理旧 LS key（可选，不影响功能）
    }
}
```

### 旧 `jxz_db` 中的历史字段清理

迁移完成后，从 `jxz_db` 中删除：
- `snapshot` key（已移至 `jxz_snapshot_db`）
- `lastTurnCommitted` key（废弃）

对应 `_migrateFromLocalStorage()` 中的迁移代码段也可同步删除。

# 姬侠传 历史管理方案

## 一、现状与问题

### 1.1 当前机制（Phase 1/2 的滑动窗口）

每轮 AI 回复解析出 `<SUMMARY>` 后写入 `summaryHistory[]`，构建 prompt 时注入为 `<RecentHistory>` 块。当记录超出 token 预算时，直接丢弃最旧的条目。

**核心缺陷**：被截断的旧事件**永久丢失**，模型无法再感知。对于有多条隐藏剧情线的游戏（如 NPC 好感度发展、未解决的悬念），截断会导致剧情连贯性下降。

### 1.2 对比开源方案

| 方案 | 优点 | 缺点 | 适合本项目？ |
|------|------|------|-------------|
| MemGPT / Letta | 自动记忆管理，理论完备 | 需要服务端进程，不兼容静态部署 | ❌ |
| Mem0 | 结构化记忆提取 | 同上，需后端服务 | ❌ |
| LangChain / LlamaIndex | 完整 RAG 框架 | 后端 Python，无法运行在浏览器/APK | ❌ |
| Transformers.js（本地 embedding） | 无服务端依赖 | 模型文件 20-50MB，首次下载慢，移动端不友好 | ⚠️ 备选 |
| **硅基流动 embedding API** | 纯 API 调用，零本地文件，中文效果好 | 需用户配置 Key | ✅ **采用** |

**结论**：采用硅基流动的 `BAAI/bge-m3` embedding API 做语义向量化召回，未配置或 API 失败时降级为现有截断方案。

---

## 二、整体架构

```
用户执行行动
  ↓
pipeline.runTurn()
  ├── [请求阶段]  构建 prompt → API 请求（无副作用）
  └── [提交阶段]  解析 → 更新状态 → 渲染 → 存档
                    ↓（异步，不阻塞）
             新 SUMMARY 调 embedding API
             → 向量存入 IndexedDB embeddings store

下一轮构建 prompt 时：
  1. 计算 query embedding（用户输入 + 最近 2 条 SUMMARY）
  2. 内存中余弦相似度，Top-K 召回（排除已在滑动窗口内的条目）
  3. 拼接为 <RecentHistory> 块注入 prompt
     - 向量召回部分：[相关历史回忆]
     - 滑动窗口部分：[近期经历]
  4. embedding 不可用时：完全跳过召回，走现有截断逻辑
```

---

## 三、不影响 SR/ST 链路的保证

- 所有 embedding 逻辑仅在 `!getRenderFunction()`（独立模式）下执行
- prompt-builder 的召回注入分支仅在 `embeddingEnabled && isStandalone` 时激活
- SR/ST 模式不加载 embedding 相关模块，不产生任何额外请求

## 四、分阶段实施计划

### Phase 3.1 — Embedding 服务层 + 向量存储

**目标**：建立 embedding 能力的基础设施，能对文本计算向量并持久化存储。

#### 4.1.1 新建 `module/embedding-service.js`

模式：`var embeddingService = (function(){...})()` IIFE，与现有模块保持一致。

| 功能 | 说明 |
|------|------|
| 调用硅基流动 `/v1/embeddings` | 格式兼容 OpenAI Embeddings API |
| 批量 embed | 多条文本一次请求（节省 API 调用次数），返回 `Float32Array[]` 或 null |
| 多 Key 轮询 | 支持逗号/分号/竖线分隔多个 Key，每次请求自动轮询（均摊免费 Key 的 QPS 限额） |
| AbortController 超时 | 15 秒超时，使用 `AbortController` + `setTimeout` 精确中断，失败返回 null |
| Engine Fingerprint | `getFingerprint()` 返回 `model + '@siliconflow'`；用户切换模型后旧向量可识别需重建 |
| 存储失败节流 | `canNotifyStoreFail()` — 10 秒冷却，防止连续弹窗 |
| 配置项 | localStorage key `jxz_embeddingConfig`：endpoint / apiKey（支持多 Key）/ model / enabled 开关 |
| 测试接口 | `testConnection()` — 发送单条"测试连接"探针，返回 `{ok, dims, error}` |

**请求格式**：
```json
POST /v1/embeddings
{
  "model": "BAAI/bge-m3",
  "input": ["文本1", "文本2"],
  "encoding_format": "float"
}
→ { "data": [{ "embedding": [0.1, 0.2, ...], "index": 0 }, ...] }
```

**对外接口**：`getConfig` / `updateConfig` / `isEnabled` / `getFingerprint` / `embed` / `canNotifyStoreFail` / `testConnection`

#### 4.1.2 改造 `module/storage-service.js` — embedding CRUD

**改动点 1**：在 `clearSummaryHistory` 函数之后插入 4 个 embedding 方法，用同一 IDB keyval store，key 格式为 `emb_<summaryId>`：

| 方法 | 说明 |
|------|------|
| `saveEmbedding(id, vector, metadata)` | 写入 `_cache` 并异步写 IDB；vector 字段存 Float32Array（IDB 结构化克隆原生支持） |
| `loadAllEmbeddings()` | 遍历 `_cache`，筛出 `emb_` 前缀条目并返回 |
| `deleteEmbedding(id)` | 从 `_cache` + IDB 删除 |
| `clearEmbeddings()` | 批量清空所有 `emb_` 条目，新游戏时调用 |

embedding 记录字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 对应 summaryHistory 条目的 id |
| vector | Float32Array | bge-m3 输出 1024 维向量（IDB 直接存，无需转 ArrayBuffer） |
| text | string | 原始 summaryText（供调试展示） |
| week | number | 对应的游戏周数 |
| createdAt | number | 时间戳 |
| fingerprint | string | 生成时的 Engine Fingerprint |

**改动点 2**：扩展 `buildSavePayload`，新增 `embeddings` 字段（Float32Array → `Array.from()` 序列化为 number[] 保证 JSON 兼容）

**改动点 3**：扩展 `importSavePayload`，恢复 embeddings（number[] → Float32Array → `saveEmbedding`），完成后调 `memoryRecall.clearCache(); memoryRecall.init()`

**改动点 4**：在 return 对象中加入 `saveEmbedding` / `loadAllEmbeddings` / `deleteEmbedding` / `clearEmbeddings`

**向量序列化工具函数**（定义在 `memory-recall.js` 中，此处备注用途）：
```javascript
// Float32Array → ArrayBuffer（存档导出时用）
function float32ToBuffer(arr) {
    return arr.buffer.slice(arr.byteOffset, arr.byteOffset + arr.byteLength);
}
// ArrayBuffer → Float32Array（导入时用）
function bufferToFloat32(buffer) {
    return new Float32Array(buffer);
}
```

#### 4.1.3 新建 `module/memory-recall.js` — 召回引擎

模式：`var memoryRecall = (function(){...})()` IIFE，依赖 `embeddingService` 和 `storageService`。

**内存缓存**：`_cache[]` — 启动时由 `init()` 从 `storageService.loadAllEmbeddings()` 填充，查询时直接遍历（数百条 × 1024 维 float32 ≈ 数 MB，可接受）。

**主接口**：
```javascript
recallRelevantMemories(queryText, topK, excludeIds, maxTokens)
// topK 默认 15，maxTokens 默认 3000
// → [{ id, text, week, similarity }, ...]
```

**实现逻辑**：
1. `embeddingService.embed([queryText])` 获取 query 向量
2. 遍历 `_cache`，跳过 excludeIds 和 fingerprint 不匹配的条目
3. 计算余弦相似度，过滤 < 0.5 的条目
4. 按相似度降序排列
5. 双重截断：累计 token ≤ maxTokens 且条数 ≤ topK
6. 失败时 10 秒冷却节流弹窗，返回 `[]`

**Token 估算**（用于双重截断）：
```javascript
function estimateTokens(text) {
    if (!text) return 0;
    var chinese = (text.match(/[\u4e00-\u9fff]/g) || []).length;
    var other = text.length - chinese;
    return Math.ceil(chinese + other / 4);
}
```

**余弦相似度**（纯 JS，无依赖）：
```javascript
function cosineSimilarity(a, b) {
    var dot = 0, normA = 0, normB = 0;
    for (var i = 0; i < a.length; i++) {
        dot += a[i] * b[i];
        normA += a[i] * a[i];
        normB += b[i] * b[i];
    }
    var denom = Math.sqrt(normA) * Math.sqrt(normB);
    return denom > 0 ? dot / denom : 0;
}
```

**对外接口**：`init` / `addToCache` / `clearCache` / `recallRelevantMemories` / `getStats` / `float32ToBuffer` / `bufferToFloat32`

### Phase 3.2 — Pipeline 集成 + Prompt 注入

**目标**：在现有流程中接入 embedding 存储和召回，不影响原有链路。

#### 4.2.1 改造 `module/pipeline.js` — 两处改动

**改动点 1**：`runTurn()` 中，`promptBuilder.buildMessages(...)` 调用之前，插入向量召回：

```javascript
// query = 用户输入 + 最近 2 条 SUMMARY
// excludeIds = 全部 summaryHistory 的 id（避免和滑动窗口重复）
var recalledMemories = [];
if (embeddingService.isEnabled()) {
    try {
        var recentContext = summaryHistoryService.getAll().slice(-2)
            .map(s => s.summaryText).join('\n');
        var queryText = preprocessed + (recentContext ? '\n' + recentContext : '');
        var excludeIds = summaryHistoryService.getAll().map(s => s.id);
        recalledMemories = await memoryRecall.recallRelevantMemories(queryText, 15, excludeIds, 3000);
    } catch(e) { recalledMemories = []; }
}
// 将 recalledMemories 传入 buildMessages
```

**改动点 2**：`_commitResponse()` 的 `console.log('[Pipeline] 提交完成')` 之后，fire-and-forget 异步存储：

```javascript
// 取最后一条新 SUMMARY，异步 embed 并存储
if (embeddingService.isEnabled() && newSummaryEntry) {
    embeddingService.embed([newSummaryEntry.summaryText])
        .then(vectors => {
            if (!vectors?.[0]) { /* 节流弹窗提示 */ return; }
            storageService.saveEmbedding(newSummaryEntry.id, vectors[0], { ... });
            memoryRecall.addToCache({ ... });
        })
        .catch(e => { /* 节流弹窗提示 */ });
}
```

#### 4.2.2 改造 `module/prompt-builder.js` — 三处改动

**改动点 1**：新增 `_buildRecalledMemoriesBlock(recalledMemories)` 函数，生成 `<RecalledMemories>` XML 块（格式与 `<HistorySummary>` 相同，每条一个时间戳 + 文本）

**改动点 2**：`_buildMsg1System` 签名增加第 4 参数 `recalledMemoriesBlock`，在 HistorySummary 之前注入：

**改动点 3**：`buildMessages()` 中：
- 从 `params.recalledMemories` 读取召回结果

## 五、降级策略

| 条件 | 行为 | 用户感知 |
|------|------|---------|
| 用户未配置 embedding | 完全跳过 | 与现有体验一致，无变化 |
| embedding API 调用失败（单次） | 弹窗提示用户；跳过本次存储/召回，走截断；未向量化的 SUMMARY 可事后通过"重建记忆索引"补算 | 弹窗提示一次，不重复打扰 |
| embedding API 超时（>3s） | 同上 | 弹窗提示一次，不重复打扰 |

**召回失败节流**（借鉴 LittleWhiteBox）：使用冷却期机制（如 10 秒），同一错误在冷却期内不重复弹窗：
```javascript
let lastFailAt = 0;
const COOLDOWN_MS = 10_000;
function canNotifyFail() {
    const now = Date.now();
    if (now - lastFailAt < COOLDOWN_MS) return false;
    lastFailAt = now;
    return true;
}
```
| IndexedDB 不可用 | 跳过向量持久化，当次内存可用 | 刷新后无历史记忆 |

## 十、后续演进路线（Phase 3.5 — Phase 5）

Phase 3 面向 ~1000 条 SUMMARY 足够好用；当重度用户积累到几千条对话时，可按需逐步升级：

### Phase 3.5 — 轻量优化（纯前端，零额外 API 调用）

**触发条件**：Phase 3 上线后发现"召回噪声增多"或"旧事件偶尔漏召"。

| 优化项 | 做法 | 效果 |
|--------|------|------|
| 时间衰减加权 | `最终分 = 相似度 × (1 - α × 归一化时间距离)`，α 可配（建议 0.05-0.1） | 越久远的记忆需要更高相似度才能入围，减少"语义近但不相关"的旧条目 |
| 相似度阈值上调 | 从 0.5 → 0.6 | 直接砍掉低质量召回 |
| NPC 名字精确匹配加分 | 维护 `Map<NPC名, summaryId[]>`，query 中出现 NPC 名时，对应 SUMMARY 加 0.1 分 | 解决"用户提到具体人名但向量没排最前"的问题 |
| Top-K 扩大 + 去重合并 | 先取 Top-30，合并连续周数的相关记录（同事件线），再截取 15 条 | 减少重复、保留事件完整性 |

### Phase 4 — 关键词倒排 + Reranker（需额外 API 调用）

**触发条件**：Phase 3.5 后仍有"语义相似但不相关"的召回污染。

| 模块 | 做法 | 成本 |
|------|------|------|
| 轻量关键词索引 | 对每条 SUMMARY 提取 NPC 名 + 地点 + 关键物件，建立 `Map<关键词, summaryId[]>` | 零 API，纯前端 |
| 两路融合（W-RRF） | 向量召回 Top-30 + 关键词命中集 → 加权倒数排序融合 → 取 Top-15 | 零额外 API |
| Reranker 精排 | 融合后 Top-15 调 `bge-reranker-v2-m3` 做二次精排 | 每轮多一次 API（rerank 请求） |

**为什么需要关键词补充向量**：向量擅长"大方向对"（语义相关），但遇到精确实体（如"钱塘君给我的那把剑"）时，关键词直接匹配"钱塘君"+"剑"比向量更准。两路融合取长补短。

### Phase 5 — LLM 关系图 + PPR 扩散（重度方案）

**触发条件**：需要"跨角色联想召回"能力（如跟 A 对话时自动召回 A 和 B 的过往恩怨）。

| 模块 | 做法 | 成本 |
|------|------|------|
| 三元组提取 | 每轮 AI 回复后，额外调一次 LLM 从 SUMMARY 提取 `{s, t, r}` 关系边 | 每轮多一次 LLM 请求（可用小模型降本） |
| 关系图存储 | IDB 新增 `edges` store，节点=角色/地点，边=互动关系+权重 | 零额外 API |
| PPR 扩散 | 从当前对话涉及的节点出发，Personalized PageRank 迭代 → 关联节点的记忆提权 | 纯 JS 计算（~50 行），零 API |

**三元组提取 prompt 参考**（借鉴 LittleWhiteBox）：
```
从一轮对话中提取关系三元组：
- s: 施事方（角色名）
- t: 受事方（角色名）  
- r: 互动行为（6-12字动作短语，如"以圣剑刺穿心脏"）
每轮 1-3 条，无明确互动时返回空数组。
```

**PPR 效果举例**：你在跟钱塘君说话 → 图中钱塘君与洞庭君关系边权高 → 扩散后洞庭君相关记忆被提权 → 即使用户没提到洞庭君，相关恩怨也会被召回注入 prompt。

### 演进路线总览

```
Phase 3（当前）── 纯向量 dense 召回 + 相似度排序 ──── 适用 ~1000 条
    ↓
Phase 3.5 ───── 时间衰减 + NPC精确匹配加分 ────── 适用 ~2000 条
    ↓
Phase 4 ─────── 关键词倒排融合 + Reranker ─────── 适用 ~5000 条
    ↓
Phase 5 ─────── LLM三元组 + 关系图 + PPR扩散 ──── 无上限
```

每个 Phase 独立可选，按实际召回效果决定是否升级。

# 周总结优化方案（v2）

## 目标

在每次新周开始（`newWeek=1`）时，从当周的完整 uiConversation 原始正文中，异步调一轮独立 LLM 请求，生成一份高质量的周总结，替换 weekHistory 中的初版总结。

---

## 核心设计要点

1. weekHistory 条目增加 `source` 字段：`'runTurn'`（初版，来自主 LLM 回复的 SUMMARY）或 `'runSummary'`（优化版，来自独立总结请求）
2. **一旦某条 weekHistory 的 source 为 `'runSummary'`，视为最终版本**——重生成时不会覆盖它，也不会重新触发总结
3. summaryBuff 和 markWeekUiIndex 存储在 IDB（通过 storageService），存档包含 summaryBuff

---

## 核心数据流

```
newWeek=1 触发，currentWeek 已自增
    │
    ├─ [检查] weekHistory 中是否已存在 markWeek===currentWeek 且 source==='runSummary' 的条目？
    │    ├─ YES → 跳过全部（不写初版、不收集 buff、不触发 runSummary）
    │    └─ NO → 继续 ↓
    │
    ├─ [1] 读 oldIdx = storageService.getMarkWeekUiIndex()
    ├─ [2] 收集 summaryBuff：uiConversation.slice(oldIdx) 中 assistant 条目拼接
    ├─ [3] 持久化 summaryBuff = { targetMarkWeek: currentWeek, prevMarkWeekUiIndex: oldIdx, text }
    ├─ [4] weekHistoryService.append(parsed.summaries, 'runTurn')  ← 初版总结写入
    ├─ [5] storageService.setMarkWeekUiIndex(uiConversation.length) ← 与步骤4同时
    │
    └─ [异步，_commitResponse 完成后]
         └─ setTimeout → summaryRunner.scheduleSummary()
              │
              └─ runSummary(buff)
                   │
                   ├─ 先检查：weekHistory 已有 runSummary 条目？→ 清空 buff，返回
                   ├─ 调 API 成功 → replaceByMarkWeek(targetMarkWeek, result, 'runSummary')
                   │                 → 清空 summaryBuff
                   └─ 调 API 失败 → buff 保留，延迟重试
```

---

## weekHistory 条目结构

```javascript
{
    id: 'w1716..._abc123',
    week: currentWeek,          // 写入时的 currentWeek
    markWeek: currentWeek,      // 用于 replaceByMarkWeek 定位
    summaryText: '...',         // 总结正文 + 后缀
    source: 'runTurn',          // 'runTurn' | 'runSummary'
    createdAt: Date.now()
}
```

`source` 的值：
- `'runTurn'`：初版，来自主 LLM 回复中的 `<SUMMARY>` 块
- `'runSummary'`：优化版，来自独立总结请求，**视为最终版本，不再覆盖**

## 新增状态字段

### 存储位置：IndexedDB (jxz_db) + localStorage 兼容层（Write-Through Cache）

| storageService key | IDB/cache key | localStorage key | 含义 |
|---|---|---|---|
| `markWeekUiIndex` | `markWeekUiIndex` | `jxz_markWeekUiIndex` | 上次写入初版总结时 uiConversation 的长度，初始值 `0` |
| `summaryBuff` | `summaryBuff` | `jxz_summaryBuff` | 待总结的正文（JSON 对象）|

`summaryBuff` 结构：
```json
{
  "targetMarkWeek": 5,
  "prevMarkWeekUiIndex": 42,
  "text": "（拼接的 assistant 正文，换行分隔）"
}
```

- `targetMarkWeek`：currentWeek（已自增的新周数），与 weekHistory 条目的 `markWeek` 一致
- `prevMarkWeekUiIndex`：写入前的旧 markWeekUiIndex 值，重生成时用于恢复

### 存档包含 summaryBuff

`createSave()` payload 加入 `summaryBuff`；`loadSave()` 后恢复到 storageService，`resumeOnLoad()` 自动检查并触发重试。

## 重生成场景分析

### 场景A：截断/中止后重生成（lastTurnCommitted = false）

步骤 ⑧-⑬ 在 `if (summaries.length > 0)` 块内，截断时整体不执行：
- weekHistory 未写入 → 无需回滚
- summaryBuff 未写入 → 无 stale buff
- markWeekUiIndex 未更新 → 下次 slice 仍正确
- regenerate 检测 `lastTurnCommitted=false` → 跳过 pop 逻辑

**天然安全，无隐患。**

### 场景B：生成成功、runSummary 已完成后重生成

weekHistory 条目 source 已为 `'runSummary'`（最终版本）。

```
_confirmRegenerate：
  - _popFromWeek=true，检查 weekHistory 末尾条目
  - source === 'runSummary' → 不 pop weekHistory（保留最终版本）
  - 仅 popLastAssistantUI()
  - clearSummaryBuff()（如果有的话）
  - 不恢复 markWeekUiIndex（已是正确值）

_commitResponse（重生成后）：
  - 步骤 ⑧ 检查：已存在 runSummary 条目 → 跳过 ⑨-⑬
  - 正常执行 ⑭-⑯（写 assistant UI、标记 committed）
  - ⑰ 触发 scheduleSummary → runSummary 发现已有 runSummary 条目 → clearBuff，return
```

**结论：runSummary 完成后重生成完全安全，最终版本总结不被覆盖。**

### 场景C：生成成功、runSummary 尚未完成时重生成

weekHistory 条目 source 仍为 `'runTurn'`（初版）。

```
_confirmRegenerate：
  - _popFromWeek=true，检查 weekHistory 末尾条目
  - source === 'runTurn' → pop weekHistory
  - popLastAssistantUI()
  - 从 summaryBuff.prevMarkWeekUiIndex 恢复 markWeekUiIndex
  - clearSummaryBuff()（取消旧 runner 的写入目标）

_commitResponse（重生成后）：
  - 步骤 ⑧ 检查：无 runSummary 条目 → 正常流程
  - ⑨-⑬：重新收集 buff、写初版、更新 markWeekUiIndex
  - ⑰ 触发 scheduleSummary

旧 runSummary API 返回时：
  - replaceByMarkWeek(oldTarget) → 条目已被 pop → 找不到 → 静默跳过
  - getSummaryBuff().targetMarkWeek ≠ 旧 buff.targetMarkWeek → 不 clear
  - _running=false → scheduleSummary() → 捡起新 buff → 正常处理 ✓
```

**结论：安全，新 buff 最终会被正确处理。**

## 新增模块：`module/summary-runner.js`

```javascript
var summaryRunner = (function() {
    var _running = false;
    var _retryDelay = 5000;

    function isRunning() { return _running; }

    function scheduleSummary() {
        var buff = storageService.getSummaryBuff();
        if (!buff || !buff.text || !buff.targetMarkWeek) return;
        if (_running) {
            console.log('[SummaryRunner] 上次尚未完成，跳过本次调度');
            return;
        }
        setTimeout(function() { runSummary(buff); }, 0);
    }

    async function runSummary(buff) {
        _running = true;
        try {
            // 前置检查：是否已有 runSummary 最终版本
            if (weekHistoryService.hasRunSummaryEntry(buff.targetMarkWeek)) {
                storageService.clearSummaryBuff();
                console.log('[SummaryRunner] 已存在 runSummary 条目，清空 buff');
                _running = false;
                return;
            }

            // 字符上限
            var apiCfg = apiService.getConfig();
            var charLimit = Math.floor((apiCfg.maxContextTokens || 500000) * 1.5 * 0.6);
            var text = buff.text;
            if (text.length > charLimit) {
                text = text.slice(text.length - charLimit);
            }

            // 构建 user prompt（{N} = buff.turnCount，{buff内容} = 截断后的 text）
            var userPrompt = '以下是本周（共' + (buff.turnCount || '若干') + '轮）的游戏对话原文：\n\n'
                + text
                + '\n\n请将上述内容拆解为若干条事件记录，严格按格式输出，确保以</SUMMARY>结束。';

            // 调用 API
            var result = await apiService.chat([
                { role: 'system', content: SUMMARY_SYSTEM_PROMPT }, // 见 prompt 设计节
                { role: 'user',   content: userPrompt }
            ]);

            // 截断检测：</SUMMARY> 缺失则视为失败，重试
            if (!result || !result.includes('</SUMMARY>')) {
                throw new Error('SUMMARY 输出被截断，缺少 </SUMMARY>');
            }
            var summaryMatch = result.match(/<SUMMARY>([\s\S]*?)<\/SUMMARY>/);
            var summaryText = summaryMatch ? summaryMatch[1].trim() : '';
            if (!summaryText) { throw new Error('SUMMARY 内容为空'); }

            weekHistoryService.replaceByMarkWeek(buff.targetMarkWeek, summaryText, 'runSummary');
                // 校验后清空
                var currentBuff = storageService.getSummaryBuff();
                if (currentBuff && currentBuff.targetMarkWeek === buff.targetMarkWeek) {
                    storageService.clearSummaryBuff();
                    console.log('[SummaryRunner] 周总结替换成功，已清空 buff');
                } else {
                    console.log('[SummaryRunner] buff 已被更新，跳过 clear');
                }
            } else {
                throw new Error('LLM 返回为空');
            }
        } catch(e) {
            console.warn('[SummaryRunner] 总结失败，延迟重试:', e.message);
            setTimeout(function() {
                _running = false;
                scheduleSummary();
            }, _retryDelay);
            return;
        }
        _running = false;
        scheduleSummary(); // 捡起可能在 running 期间写入的新 buff
    }

    function resumeOnLoad() {
        if (!_running) scheduleSummary();
    }

    return { scheduleSummary, resumeOnLoad, isRunning };
})();
```

## weekHistoryService 新增方法

```javascript
/** 检查是否已存在 runSummary 最终版本 */
function hasRunSummaryEntry(targetMarkWeek) {
    var history = load();
    for (var i = 0; i < history.length; i++) {
        if (history[i].markWeek === targetMarkWeek && history[i].source === 'runSummary') {
            return true;
        }
    }
    return false;
}

/** 按 markWeek 替换条目（找不到则静默跳过） */
function replaceByMarkWeek(targetMarkWeek, newText, newSource) {
    var history = load();
    var found = false;
    for (var i = 0; i < history.length; i++) {
        if (history[i].markWeek === targetMarkWeek) {
            var _cw = history[i].week || targetMarkWeek;
            var _suffix = '\n[至' + (_cw - 1) + '周的历史记录]';
            history[i].summaryText = newText + _suffix;
            history[i].source = newSource || 'runSummary';
            // 不更新 createdAt：保留初版写入时的时间戳
            // 用于 _confirmRegenerate 中与 summaryHistory 末尾的时间比较
            found = true;
            break;
        }
    }
    if (found) {
        save(history);
        console.log('[WeekHistory] replaceByMarkWeek 成功, markWeek=' + targetMarkWeek);
    } else {
        console.log('[WeekHistory] replaceByMarkWeek 未找到条目, markWeek=' + targetMarkWeek + ', 跳过');
    }
    return found;
}
```

## summary_buff 的内容范围

取 `uiConversation.slice(markWeekUiIndex)` 中 `role === 'assistant'` 的条目，拼接 `content`：

```
[第X周 第N轮]
（正文内容）

[第X周 第N+1轮]
（正文内容）
...
```

注意：收集发生在 **appendUIConversation(assistant) 之前**，所以本轮 assistant 正文**不在** buff 中（本轮内容已在初版 SUMMARY 里覆盖）。

**字符上限：动态计算**，基于 `apiService.getConfig().maxContextTokens`：

```javascript
var charLimit = Math.floor(apiService.getConfig().maxContextTokens * 1.5 * 0.6);
// 默认 500000 tokens → 约 450000 字符上限
```

超出时取**末尾**（最近内容更重要）。

## runSummary 的 prompt 设计

**System（`SUMMARY_SYSTEM_PROMPT` 常量）：**
```
你是游戏"瀚海归义录"的故事记录者。你的任务是将一周内的游戏对话原文，拆解为若干条具体可检索的事件记录。

【输出格式，严格遵守】
<SUMMARY>
[第x年第x月第x周]
事件描述（50字以内）

[第x年第x月第x周]
事件描述（50字以内）
</SUMMARY>

【时间标注】
从对话原文中提取该事件的游戏内时间（如"第一年二月第三周"）。若当前事件原文未明确提及，沿用本段最近出现的时间标记。

【事件描述写法】
用朴实白描的叙述句，记录原文中实际发生的事，不要抽象总结腔。
- 必须保留：正式人名/称呼、地点、关键物件/道具、具体动作
- 写清楚：谁在什么地方、对谁、做了什么、结果如何
- 有则尽量保留：情绪/态度、约定/承诺、关系变化、可供玩家日后提起的线索
- 不要为了凑足字数而编造原文没有的信息

禁止空泛写法：
- 禁止："主角与NPC交谈，关系升温。"
- 禁止："两人发生冲突，气氛剑拔弩张。"
- 禁止："主角探索了某地区，有所收获。"

合格写法：
- 合格："主角携鹿茸酒前往萧白瑚的药庐，以探病为由登门，萧白瑚接受药酒但态度疏离，临走前主角以'欠一次问诊'为借口约定下周再来，萧白瑚未明确拒绝。"
- 合格："呼延显在教场当众考校主角剑法，主角用破阵七式接下前三招，第四招被打落木剑，呼延显令其闭门三日重修基础，未给明确点评。"

【数量规则】
- 依叙事价值决定，通常每轮不超过3条，无重要内容可省略
- 场景切换、人物变化时拆分条目
- 可省略：纯机械操作（查看背包、刷新商店）、无实质互动的过场、门派内参/武林动态/陇右新闻等八卦新闻内容
```

**User（运行时拼接）：**
```
以下是本周（共{N}轮）的游戏对话原文：

{buff内容}

请将上述内容拆解为若干条事件记录，严格按格式输出，确保以</SUMMARY>结束。
```
> 其中 `{N}` = `buff.turnCount`，`{buff内容}` = 字符截断后的 `buff.text`

**截断检测（runSummary 函数中，API 返回后立即执行）：**
```javascript
// </SUMMARY> 缺失 → 视为截断失败 → 走 catch 重试
if (!result || !result.includes('</SUMMARY>')) {
    throw new Error('SUMMARY 输出被截断，缺少 </SUMMARY>');
}
var summaryMatch = result.match(/<SUMMARY>([\s\S]*?)<\/SUMMARY>/);
var summaryText = summaryMatch ? summaryMatch[1].trim() : '';
if (!summaryText) { throw new Error('SUMMARY 内容为空'); }
```

**存储说明：** `summaryText` 字段只存 `<SUMMARY>` 内层正文（多条 `时间: ...\n事件描述` 结构），不含外层标签。

## 已确认方案汇总

| 项目 | 决策 |
|---|---|
| markWeekUiIndex 初始值 | `0`，第一周总结取全部 uiConversation 历史 |
| markWeekUiIndex 更新时机 | weekHistory 写入初版总结的瞬间（与 append 同步） |
| summaryBuff 收集时机 | markWeekUiIndex 更新之前 |
| runSummary 触发时机 | weekHistory 写入初版之后（异步 setTimeout） |
| summary_buff 字符上限 | 动态：`maxContextTokens × 1.5 × 0.6`，超出取末尾 |
| runSummary API 配置 | 直接复用 `apiService` |
| 失败重试机制 | 自动延迟 5s 重试，buff 保留在 IDB，直到成功或发现已有 runSummary 条目后才清空 |
| summaryBuff 存储位置 | IDB + localStorage（Write-Through Cache），存入存档 |
| weekHistory 新增字段 | `source: 'runTurn' \| 'runSummary'` |
| runSummary 完成后重生成 | 不覆盖最终版本，不重新触发总结 |
| 输出格式 | `<SUMMARY>...</SUMMARY>` 标签包裹，每事件一条（`[第x年第x月第x周]` + 白描描述） |
| 截断检测 | `</SUMMARY>` 缺失则视为失败，重新发起请求 |
| prompt 文案 | 已确认（白描风格 + 正负示例 + 多条事件结构） |
